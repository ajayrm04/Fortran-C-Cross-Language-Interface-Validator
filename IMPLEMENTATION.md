# Implementation Details — LLVM & Compiler Internals

## 1. How Fortran `BIND(C)` Maps to LLVM IR

When a Fortran compiler (e.g., `flang` or `gfortran`) compiles a `BIND(C)` interface, it emits LLVM IR that follows the **C ABI** rather than the native Fortran calling convention. This is what our validator verifies at the source level.

### 1.1 Calling Convention Lowering

```fortran
subroutine c_add_int(a, b, result) bind(c, name="add_int")
  integer(c_int), value :: a
  integer(c_int), value :: b
  integer(c_int), intent(out) :: result
end subroutine
```

The LLVM IR generated by `flang` would look like:

```llvm
define void @add_int(i32 %a, i32 %b, ptr %result) #0 {
entry:
  %sum = add i32 %a, %b
  store i32 %sum, ptr %result
  ret void
}
```

Key observations:
- **`value` parameters** → passed directly as `i32` (by value)
- **`intent(out)` without `value`** → passed as `ptr` (by reference, the Fortran default)
- **Function name** → uses the `name="add_int"` binding name as the LLVM symbol

### 1.2 Type Width Mapping to LLVM IR

| Fortran Type             | Canonical | LLVM IR Type | C Equivalent |
|--------------------------|-----------|--------------|--------------|
| `integer(c_int)`         | `int32`   | `i32`        | `int`        |
| `integer(c_int64_t)`     | `int64`   | `i64`        | `int64_t`    |
| `real(c_float)`          | `float`   | `float`      | `float`      |
| `real(c_double)`         | `double`  | `double`     | `double`     |
| `logical(c_bool)`        | `bool`    | `i8`         | `_Bool`      |
| `type(c_ptr)`            | `void*`   | `ptr`        | `void*`      |
| `complex(c_double_complex)` | `double_complex` | `{ double, double }` | `double _Complex` |
| `integer(c_size_t)`      | `size_t`  | `i64` (64-bit) | `size_t`  |

### 1.3 ABI Mismatch Consequences in LLVM IR

When signatures don't match, the LLVM linker **does not check function signatures** — it matches purely by symbol name. This means:

```llvm
; Fortran side (compiled by flang):
declare double @dot_product(ptr %x, ptr %y, i32 %n)  ; ← returns double? No!
; Actually emits: declare float @dot_product(...)      ; ← returns float (c_float)

; C side (compiled by clang):
define double @dot_product(ptr %x, ptr %y, i32 %n) {  ; ← returns double
  ...
}
```

The linker happily links these. At runtime:
- The caller expects a `double` (8 bytes) in the return register
- The callee puts a `float` (4 bytes) in the return register
- **Result:** garbage upper bits, wrong value

### 2.2 Fortran Parser — Continuation Line Merging

Fortran free-form allows line continuation with `&`:

```fortran
subroutine c_add_int(a, b, &
                     result) bind(c, name="add_int")
```

The `_merge_continuations()` method joins these into single logical lines before parsing, preserving original line numbers for error reporting.

### 2.3 Type Normalisation Strategy

The normaliser uses a **priority cascade**:

1. **Exact match** — lookup the full type string in the canonical table
2. **Dimension stripping** — remove trailing `(*)` or `(N)` and retry
3. **Prefix match** — match the longest prefix in the table

This handles variations like `real(c_double)` vs `real(c_double), dimension(*)`.

### 2.4 Validation Rules

#### Rule 1: Return Type Check
```python
if not self._types_compat(f_sig.return_type, c_sig.return_type):
    # ERROR: e.g. Fortran returns float, C returns double
```

#### Rule 2: Parameter Count Check
```python
if len(f_sig.params) != len(c_sig.params):
    # ERROR: different number of parameters
```

#### Rule 3: Per-Parameter Type Check
```python
for i in range(n):
    if not self._types_compat(fp.base_type, cp.base_type):
        # ERROR: type mismatch at position i
```

#### Rule 4: Passing Mode Check
```python
expected_c_mode = self._expected_c_mode(fp)  # Fortran VALUE → C VALUE; else → POINTER
if expected_c_mode != cp.passing_mode:
    # ERROR: e.g. Fortran passes by value, C expects pointer
```

### 2.5 Type Compatibility with Synonyms

The `_types_compat()` method goes beyond strict equality. It recognises synonym sets:

```python
synonyms = [
    {'int32', 'int', 'long'},       # platform-dependent equivalences
    {'int64', 'long long'},
    {'float', 'real'},
    {'void', 'subroutine'},
    {'char', 'character'},
]
```

It also handles pointer compatibility: `void*` is compatible with any typed pointer.

## 3. How This Relates to a Full LLVM-Based Solution

Our regex-based approach approximates what a full LLVM-based tool would do at the IR level:

| Aspect                  | Our Tool (Source-Level)         | LLVM-Based Tool (IR-Level)         |
|-------------------------|---------------------------------|------------------------------------|
| **Input**               | `.f90` + `.h` source files      | `.ll` / `.bc` LLVM bitcode         |
| **Parsing**             | Regex patterns                  | Clang AST + Flang AST             |
| **Type resolution**     | Canonical table lookup          | LLVM type system (`i32`, `double`) |
| **Macro handling**      | ❌ Not supported                | ✅ Fully expanded by preprocessor  |
| **Typedef resolution**  | ❌ Not supported                | ✅ Resolved by Clang               |
| **Accuracy**            | High for `BIND(C)` subset      | Complete for all C constructs      |
| **Dependencies**        | Python 3.7+ only               | LLVM + Clang + Flang toolchain     |
| **Portability**         | ✅ Any platform                 | Requires LLVM build                |

The key insight is that `BIND(C)` interfaces are **intentionally simple** — they use `iso_c_binding` types that map directly to C types. This regularity makes regex-based parsing viable and accurate for this specific use case.
